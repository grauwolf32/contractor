## 5. Cryptographic Functions
| Path | Function | Risk note |
|------|----------|-----------|
| `src/services/crypto.ts` | `getKey()` — SHA-256 of `CARD_ENC_KEY`; `encryptPan()` — AES-256-CBC; `decryptPan()` — AES-256-CBC; `last4()` | **HIGH**: `KEY_FALLBACK` is hardcoded (`"vault-pay-default-card-encryption-key-change-me-32"`). If `CARD_ENC_KEY` is unset, all PAN encryption uses this known key. No key rotation mechanism. |
| `src/services/auth.ts` | `hashPassword()` — bcrypt (cost 10); `verifyPassword()` — bcrypt compare; `issueAccessToken()` — JWT sign; `decodeAccessToken()` — JWT verify; `issueChallengeToken()` — JWT sign; `decodeChallengeToken()` — JWT verify; `verifyTotp()` — TOTP via `otplib` | **MEDIUM**: `JWT_SECRET` and `CHALLENGE_SECRET` have hardcoded fallback defaults. JWT TTL: 24h access, 5m challenge. bcrypt cost 10 is acceptable but could be higher. |
| `src/routes/webhooks.ts` | `expectedSignature()` — HMAC-SHA256 using `WEBHOOK_SECRET` | **MEDIUM**: `WEBHOOK_SECRET` has hardcoded fallback. In dev mode, unsigned webhooks are accepted. |
| `src/routes/cards.ts` | Imports `encryptPan`, `decryptPan`, `last4` from crypto.ts | **LOW**: PAN exposed in response body on GET `/full` endpoint. |

## 6. Security Functions
| Path | Function | Risk note |
|------|----------|-----------|
| `src/middleware/auth.ts` | `requireAuth` — extracts user from `Authorization` header via `authService.userFromAuthHeader()` | **Medium** — Relies on `authService` for token validation. No rate limiting or brute-force protection. |
| `src/middleware/errors.ts` | `errorHandler` — sanitizes internal details, returns generic `{ error: message }` JSON | **Low** — Good practice of not leaking stack traces. However, `console.error(err)` dumps full error objects to stdout, potentially leaking sensitive data. |
| `src/middleware/requireAdmin.ts` | `requireAdmin` — checks `X-Admin: 1` header | **High** — Admin check is header-based with no cryptographic verification. A client can spoof `X-Admin: 1` to bypass admin checks. |
| `src/routes/auth.ts` | Auth routes: `/register`, `/login`, `/2fa/verify` | **Medium** — bcrypt salt=10 is acceptable but on the low side (12+ recommended). No account lockout or rate limiting on login. TOTP 2FA is opt-in, not enforced. |

**Missing security features:** No CSP headers, no rate limiting, no CSRF protection, no HTTPS enforcement in app code, no sandboxing middleware.

## Security Risk Summary
| Risk level | Paths | Rationale |
|-----------|-------|-----------|
| CRITICAL | `src/services/crypto.ts:3,6` | Card PAN encryption key (`CARD_ENC_KEY`) falls back to hardcoded default `"vault-pay-default-card-encryption-key-change-me-32"`. If not set in production, all card PAN data is encrypted with a known key, allowing decryption of every stored card number. |
| HIGH | `src/services/auth.ts:7` | JWT signing secret (`JWT_SECRET`) falls back to `"vault-pay-dev-jwt-secret"`. Attackers can forge valid JWT tokens. |
| HIGH | `src/services/auth.ts:11` | 2FA challenge secret (`TWO_FACTOR_CHALLENGE_SECRET`) falls back to `"vault-pay-2fa-challenge"`. Predictable 2FA challenge tokens. |
| HIGH | `src/routes/webhooks.ts:8` | Webhook HMAC secret (`PAYMENT_WEBHOOK_SECRET`) falls back to `"vault-pay-payment-webhook-dev-secret"`. Webhook signature verification bypassable. |
| HIGH | `src/middleware/requireAdmin.ts` | Admin authorization uses `X-Admin: 1` header with no cryptographic verification. Any client can spoof admin access. |
| MEDIUM | `src/db/schema.sql` | Plaintext storage of `password_hash`, `totp_secret`, and `pan_encrypted` in SQLite. |
| MEDIUM | `src/services/auth.ts` | bcrypt cost factor 10 is on the low side (12+ recommended). |
| MEDIUM | `src/routes/auth.ts` | No rate limiting on registration, login, or 2FA endpoints — vulnerable to brute-force attacks. |
| MEDIUM | `src/routes/auth.ts` | TOTP 2FA is opt-in, not enforced for all users. |
| MEDIUM | Entire app | No Content-Security-Policy (CSP) headers configured. |
| MEDIUM | `src/middleware/errors.ts` | `console.error(err)` dumps full error objects to stdout, potentially leaking sensitive data. |
| MEDIUM | `src/services/auth.ts` | Both `JWT_SECRET` and `CHALLENGE_SECRET` have hardcoded fallback defaults. |
| MEDIUM | `src/routes/webhooks.ts` | `WEBHOOK_SECRET` has hardcoded fallback; unsigned webhooks accepted in dev mode. |
| LOW | `src/db/connection.ts` | SQLite DB in `data/vault-pay.db` with no encryption at rest configured. |
| LOW | `src/routes/cards.ts` | Full card PAN decrypted and exposed in response body on `/full` endpoint. |
| LOW | Root | No lock files for dependency pinning — dependency versions not pinned. |
| LOW | Entire app | No HTTPS enforcement in application code (should be handled by gateway). |
